# Introduction

It's no problem when the data shape is clear(e.g. [xs ,ys]), but when we are in experiment, and have a demand to change the shape(e.g. [xs, xs, ys] or [xs, ys, ys]), we may need to extends a subclass or rebuild a class to do this. I don't think it's a good method.

So, in this post, the class DatasetBuilder can help shape the data flexibly, and avoid create extra classes.

DatasetBuilder divide all data into three class:

  1. index
  2. input
  3. label

Based on these three classes, DatasetBuilder provides a flexibly construct method.

# Get start








 
 
 
 
 
 
 
 

from thexp import DatasetBuilder
from torchvision.datasets import CIFAR10

_dataset = CIFAR10(...)
xs, ys = _dataset.data, _dataset.targets

transform = ...
dataset = (
    DatasetBuilder()
        .add_inputs(xs).add_labels(ys) # add data source
        .toggle_id() # add index
        .add_x(transform=transform) # add input with transform `transform`
        .add_x(transform=transform) # add another input with transform `transform`
        .add_y() # add label
)

# Multi-source

Construct multi-source dataset can also be implemented in few code.







 
 







xsa, ysa = _dataset.data, _dataset.targets
xsb, ysb = _dataset.data, _dataset.targets

transform = ...
dataset = (
    DatasetBuilder()
        .add_inputs(xsa).add_labels(ysa) # default source name is `xs`, `ys`
        .add_inputs(xsb, source_name='bxs').add_labels(ysb, source_name='bys') # All source must have different source name.
        .toggle_id()
        .add_x(transform=transform)
        .add_x(transform=transform, source='bxs') # add another input from source `bxs`
        .add_y()
        .add_y(source='bys') # add another label from source `bys`
)

The size of dataset can be virtual expand. For example, there is a dataset only have 100 samples, and the batch size is larger than it(e.g., 128, 256). So, the mathod .virtual_sample(size) can be called to virtually expand the size of dataset.









 

batch_size = 128
dataset = (
    DatasetBuilder(xs, ys)
        .toggle_id()
        .add_x()
        .add_y()
        .subset(train_idx)
        .virtual_sample(batch_size * (len(train_idx) // batch_size + 1))
)

# Delegate

DatasetBuilder samples data by index, so it can't create a dataset that sample data by other factors. For example, In face recognition, a normal loss is triplet loss, it needs three samples per batch: anchor sample, positive sample, and negtive sample, and they have different index in the dataset.

To fill this gap, a Delegate is provided. It receives the index, and you can construct its details and return the samples you want.

from thexp import Delegate
class RandomDelegate(Delegate):
    def __init__(self, xs, ys):
        super().__init__()
        self.xs, self.ys = xs, ys

    def __len__(self):
        return len(self.xs)

    def __call__(self, index, builder=None):
        pos_x = self.xs[index]
        neg_x = self.xs[np.random.randint(0,len(self.xs))]

        return self.X_(pos_x), self.X_(neg_x)


_dataset = CIFAR10(...)
xs, ys = _dataset.data, _dataset.targets
RandomDelegate(xs, ys)

dataset = (
    DatasetBuilder(xs, ys)
        .toggle_id()
        .add_x()
        .add_delegate(RandomDelegate(xs, ys), transform=...)
)

Data order

There are two order rules, first, sort by main class (index, input, label), and then, sort by call order.

And data provided by Delegate will always be the last one.

So, the two code fragment below have the same order

.add_x()
.add_delegate(RandomDelegate(xs, ys), transform=...)
